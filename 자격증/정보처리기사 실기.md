# 1장 : 요구사항 확인

## 소프트웨어 생명 주기

소프트웨어 생명 주기란 소프트웨어를 개발하기 위한 설계, 운용, 유지보수 등의 과정을 각 단계별로 나눈 것이다.

- 폭포수 모형 : 이전 단계로 돌아갈 수 없다는 전제 하에 각 단계를 확실히 매듭짓고, 그 결과를 철저히 검토 및 승인 후 다음 단계로 진행하는 개발 방법론
- 프로토타입 모형 : 사용자의 요구사항을 파악하기 위해 실제 개발될 소프트웨어에 대한 견본품을 만들어 최종 결과물을 예측하는 모형
- 나선형 모형 : 나선을 따라 돌듯 여러 번의 소프트웨어 개발 과정을 거쳐 점진적으로 완벽한 최종 소프트웨어를 개발하는 모형
    - 보헴이 제안하였으며, 위험 분석 기능을 추가한 모형이기에 계획 수립 → 위험 분석 → 개발 및 검증 → 고객 평가 순서대로 돌아간다.
- 애자일 모형 : 고객의 요구사항 변화에 유연하게 대응할 수 있도록 일정한 주기를 반복하며 개발하는 모형
    - 빠르고 낭비 없이 만들기 위해, 고객과의 소통 중심
    - 대표적으로 스크럼, XP, 칸반, Lean, FDD(기능 중심 개발) 등이 있다.
    - 상호작용, 실행되는 SW, 고객과 협업, 변화에 반응에 더 가치를 둔다.
- 소프트웨어 공학(SE) : 소프트웨어의 위기를 극복하기 위한 방안으로 연구된 학문
    - 여러 방법론, 도구, 관리 기법을 통해 소프트웨어 품질과 생산성 향상

## 스크럼

스크럼은 팀이 중심이 되어 개발의 효율성을 높이는 기법이다.

- 스크럼 팀은 제품 책임자, 스크럼 마스터, 개발 팀으로 구성된다.
- 2~4주간의 스프린트를 수행하고, 매일마다 일일 스크럼 회의를 진행한다.
- 계획하여 진행(스프린트)한 후, 회의와 검토를 거쳐 회고한다.
- 제품 백로그 : 제품 개발에 필요한 모든 요구사항을 적고, 우선순위에 따라 정렬한 것
- 소멸 차트(Burn-down Chart) : 작업의 진행 상황 확인을 위해 시간의 경과에 따라 남은 시간 그래프로 표현한 차트

## XP(eXtreme Programming)

XP는 수시로 발생하는 고객의 요구사항에 유연하게 대응, 고객의 참여와 개발 과정의 반복을 극대화하여 개발 생산성을 향상시키는 방법론이다.

- XP의 5가지 핵심 가치
    1. 의사소통
    2. 단순성
    3. 용기
    4. 존중
    5. 피드백
- XP의 주요 실천 방법
    - Pair Programming : 다른 사람과 함께 수행, 책임을 공동으로 나눠 가짐
    - Collective Ownership : 개발 코드에 대한 권한과 책임을 공동으로 소유
    - Continuous Integration : 개발된 코드들은 하나의 작업이 마무리되면 지속적 통합
    - Refactoring : 프로그램의 기능 변경 없이 시스템을 재구성. 프로그램을 쉽게 이해하고 쉽게 수정하여 빠르게 개발할 수 있도록 하기 위함

## 요구사항 정의

요구사항은 소프트웨어가 어떤 문제를 해결하기 위해 제공하는 서비스에 대한 설명과, 정상적으로 운영되는데 필요한 제약조건이다.

- 요구사항은 기능/비기능 요구사항과 사용자/시스템 요구사항으로 구분된다.
- 기능 요구사항 : 시스템이 무엇을 하고 어떤 기능인지, 기능이나 수행과 관련된 요구사항
- 비기능 요구사항 : 품질이나 제약사항과 관련된 요구사항으로, 성능, 인터페이스, 보안, 품질 등
- 사용자 요구사항 : 사용자 관점에서 본 시스템이 제공해야 할 요구사항, 친숙한 표현
- 시스템 요구사항 : 개발자 관점에서 본 시스템 전체가 사용자와 다른 시스템에 제공해야 할 요구사항, 전문적이고 기술적인 용어이며 소프트웨어 요구사항이라고도 함

## 요구사항 개발 프로세스

요구사항 개발 프로세스는 개발 대상에 대한 요구사항을 체계적으로 도출하고 분석한 후 명세서에 정리한 다음 확인 및 검증하는 일련의 구조화된 활동이다.

- 도출 → 분석 → 명세 → 확인
- 요구사항 도출 : 시스템, 사용자, 개발자 등 시스템 개발에 관련된 사람들이 서로 의견을 교환하며 요구사항을 어떻게 수집할 것인지 식별하고 이해하는 과정
    - 브레인스토밍, 프로토타이핑, 유스케이스, 워크샵, 설문, 청취 및 인터뷰
- 요구사항 분석 : 개발 대상에 대한 사용자의 요구사항 중 명확하지 않거나 모호하여 이해되지 않는 부분 발견 및 걸러내기
    - 자료 흐름도(DFD), 자료사전(DD)
- 요구사항 명세 : 분석된 요구사항을 바탕으로 모델을 작성하고 문서화하는 것
- 요구사항 확인 : 개발 자원을 할당하기 전, 요구사항 명세서가 정확하고 안전한지 검토
- 요구공학 : 요구사항을 정의하고, 분석 및 관리하는 프로세스를 연구하는 학문
- 정형 명세 기법 : 수학적 원리, 모델 기반, 간결하고 정확하게
- 비정형 명세 기법 : 상태, 기능, 객체 중심으로 자연어를 기반으로 서술, 다이어그램

## 개발 기술 환경 파악

개발하고자 하는 소프트웨어와 관련된 운영체제, DBMS, 미들웨어 등을 선정할 때 고려해야할 사항

- 운영체제 : 컴퓨터 시스템의 자원을 효율적으로 관리, 사용자가 컴퓨터를 편리하고 효율적으로 사용할 수 있도록 환경을 제공하는 소프트웨어
- 데이터베이스 관리 시스템(DBMS) : 사용자와 데이터베이스 사이에서 사용자의 요구에 따라 정보를 생성하고, 데이터베이스를 관리해 주는 소프트웨어
- 웹 애플리케이션 서버(WAS) : 사용자의 요구에 따라 변하는 동적인 컨텐츠를 처리하는 미들웨어
- 오픈소스 : 별다른 제한 없이 누구나 사용할 수 있도록 소스코드를 공개한 소프트웨어

## 요구사항 분석

요구사항 분석은 소프트웨어 개발의 실제적인 첫 단계로, 개발 대상에 대한 사용자의 요구사항을 이해하고 문서화하는 활동을 의미한다.

- 구조적 분석 기법 : 자료의 흐름과 처리를 중심으로 하는 요구사항 분석 방법, 도형 중심으로 하향식 방법을 사용하여 시스템 세분화
- 자료 흐름도(DFD)
    - 자료의 흐름 및 변환 과정과 기능을 도형 중심으로 기술하는 방법
    - 프로세스(원), 자료흐름(화살표), 자료저장소(선), 단말(사각형)
- 자료 사전 (DD)
    - 자료 흐름도에 있는 자료를 더 자세히 정의하고 기록한 것
    - 자료의 생략( ), 자료의 선택[ | ], 자료의 반복{ }

## CASE (Computer Aided Software Engineering)

소프트웨어 개발 과정 중 사용되는 요구분석, 설계, 구현, 테스트 및 디버깅 과정의 전체 혹은 일부를 컴퓨터와 전용 소프트웨어 도구를 사용해 자동화하는 도구.

요구사항을 자동으로 분석하고, 요구사항 부넉 명세서를 기술하도록 개발된 도구이다.

- 소프트웨어 생명 주기의 전 단계를 연결
- 다양한 소프트웨어 개발 모형을 지원
- 개발 단계의 표준화, 자료 흐름도 작성
- 그래픽 지원
- 대표적인 CASE
    - SADT : 시스템 정의, 소프트웨어 요구사랑 분석 등의 도구이며 SoftTech 사에서 개발

## HIPO (Hierarchy Input Process Output)

시스템의 분석 및 설계, 또는 문서화에 사용되는 기법으로, 시스템 실행 과정인 입력, 처리, 출력의 기능을 표현한 것

- 하향식 소프트웨어 개발 문서화 도구
- 가시적 도표, 총체적 도표, 세부적 도표의 HIPO Chart가 존재

## UML (Unified Modeling Language)

시스템 분석, 설계, 구현 등 시스템 개발 과정에서 시스템 개발자와 고객 간의 의사소통이 원활하게 이루어지도록 표준화한 대표적인 객체지향 모델링 언어이다.

- 럼바우, 부치, 조콥슨 등의 객체지향 방법론의 장점을 통합
- UML은 사물, 관계, 다이어그램으로 구성
- 사물 : 다이어그램 안에서 관계가 형성될 수 있는 대상들
    - 구조 사물 : 시스템의 개념적, 물리적 요소, 클래스, 유즈케이스, 컴포넌트, 인터페이스, 노드
    - 행동 사물 : 시간과 공간에 따른 요소들의 행위, 상호작용, 상태 머신
    - 그룹 사물 : 요소들을 그룹으로 묶어 표현, 패키지
    - 주해 사물 : 부가적인 설명이나 제약조건, 노트

### UML - 관계

관계는 사물과 사물 사이의 연관성을 표현하는 것이다.

- 연관 관계는 2개 이상의 사물이 서로 관련되어있다.
- 실선, 화살표로 표현하며, 양방향 관계의 경우 화살표를 생략하고 실선만 사용한다.
- 사람 1→1 집 인 경우, 한 사람당 하나의 집을 가지며, 사람은 어떤 집을 소유하는지 아나 집은 자신이 누구의 소유인지 모른다는 의미이다.
- 집합 관계는 마름모 모양의 화살표를 쓰며, 포함되는 쪽(부분) → 포함하는 쪽(전체) 이다.
- 포함 관계는 집합 관계의 특수 형태로, 포함하는 사물의 변화가 포함되는 사물에게 영향을 미치는 관계이다. 마름모 모양의 화살표가 검정색이다. 문 ← 열쇠
- 일반화 관계는 하나의 사물이 다른 사물에 비해 더 일반적이거나 구체적인 관계이다.
    - 일반적인 개념을 상위(부모) 로 두고, 자식 → 부모 방향으로 화살표를 둔다.
- 의존 관계는 필요에 의해 서로에게 영향을 주는 짧은 시간 동안만 연관을 유지하는 관계이다.
    - 점선으로 표현한다.
- 실체화 관계는 사물이 할 수 있거나 해야 하는 기능을 연결해놓은 것이다.
    - 속이 빈 점선 화살표를 사용해 행위 → 주체로 연결한다.

### UML - 다이어그램

다이어그램은 사물과 관계를 도형으로 표현한 것이다.

- 구조적 다이어그램(정적) : 클래스 다이어그램, 객체 다이어그램, 컴포넌트 다이어그램, 배치 다이어그램, 복합체 구조 다이어그램, 패키지 다이어그램
- 행위 다이어그램(동적) : 유스케이스 다이어그램, 순차 다이어그램, 커뮤니케이션 다이어그램, 상태 다이어그램, 활동 다이어그램
- 스테레오 타입 : UML에서 표현하는 기본 기능 외에 추가적인 기능을 표현하는 것
    - 길러멧이라는 <<>> 겹화살괄호 사이에 표현할 형태를 기술한다

## 다이어그램

- 유스케이스 다이어그램 : 개발될 시스템을 이용해 수행할 수 있는 기능을 사용자의 관점에서 표현
    - 한 시스템의 네모 안에 다양한 기능을 넣고 사용자와 상호작용
- 활동 다이어그램 : 사용자의 관점에서 시스템이 수행하는 기능을 처리 흐름에 따라 순서대로 표현
    - 액티비티 수행을 담당하는 주체를 구분하는 선인 스윔 레인을 사용
- 클래스 다이어그램 : 클래스와 클래스가 가지는 속성, 클래스 사이의 관계를 표현한 것
- 순차 다이어그램 : 시스템이나 객체들이 메시지를 주고받으며 상호작용하는 과정을 표현
- 패키지 다이어그램 : UML 정적 모델링 중 하나로, 관련있는 요소들을 그룹화한 패키지 간의 의존 관계를 표현한 것
    - 폴더처럼 생긴 패키지 내에 객체가 있고, 의존 관계들로 연결되어있다.

## 소프트웨어 개발 방법론

소프트웨어  개발, 유지보수 등에 필요한 여러 가지 일들의 수행 방법과, 이러한 일들을 효율적으로 수행하려는 과정에서 필요한 각종 기법 및 도구를 체계적으로 정리하여 표준화한 거

- 구조적 방법론 : 사용자의 요구사항을 파악하여 문서화하는 처리 중심의 방법론
    - 분할 정복 원리를 사용한다.
- 정보공학 방법론 : 계획, 분석, 설계, 구축에 정형화된 기법들을 상호 연관성 있게 통합 및 적용하는 자료 중심의 방법론이다.
- 객체지향 방법론 : 객체들을 조립해서 필요한 소프트웨어를 구현하는 방법론이다.
- 컴포넌트 기반 방법론 : 컴포넌트를 조합하여 하나의 새로운 애플리케이션을 만드는 방법론이다.
    - 컴포넌트의 재사용이 가능해 시간과 노력 절감 가능
- 제품 계열 방법론 : 특정 제품에 적용하고 싶은 공통된 기능을 정의해 개발하는 방법론

## 비용 산정 기법

- 상향식 비용 산정 기법 : 세부적인 작업 단위별로 비용을 산정한 후 집계해 전체 비용을 산정
- 주로 LOC, 개발 단계별 인월수, 수학적 산정 기법이 있다.
    - 개발 단계별 인월수 기법은 LOC를 보완한 기법, 각 기능을 구현시키는데 필요한 노력을 생명 주기의 각 단계별로 산정

## LOC (원시 코드 라인 수) 기법

각 기능의 원시 코드 라인 수의 비관치, 낙관치, 기대치를 측정해 예측치를 구하고, 비용을 산정

- 예측치 = (낙관치 + 비관치 + 4기대치) / 6

## COCOMO (Constructive Cost Model)

LOC에 의한 비용 산정 기법, 보헴의 비용 예측 기법

- 조직형(Organic Mode) 중소 규모의 소프트웨어, 5만라인 이하
- 반 분리형(Semi-Detached Model) 중형 소프트웨어, 30만 라인 이하
- 내장형(Embedded Mode) 대형 소프트웨어, 30만 라인 이상

## Putnam

소프트웨어 생명주기의 전 과정 동안 사용될 노력의 분포도를 예상하는 모델

- SLIM : Rayleigh-Norden 곡선과 Putnam으로 개발된 자동화 추정 도구

## FP (기능 점수) 모형

소프트웨어의 기능을 증대시키는 요인별로 가중치 부여, 요인별 가중치를 합산하여 총 기능 점수를 산출하며 총 기능 점수와 영향도를 이용하여 기능 점수를 구한 후 이를 이용해 비용을 산정

- FP (Function Point)
- ESIMACS : FP 모형을 기초로 하여 개발된 자동화 추정 도구

## 프로젝트 일정 계획

- PERT : 전체 작업의 상호 관계를 표시하는 네트워크
- CPM : 임계 경로 기법이며, 작업을 나열하고 작업에 필요한 소요 기간을 예측
    - 임계 경로는 최장 경로이다.
- 간트 차트 : 프로젝트의 각 작업 시작과 종료의 작업일정을 막대 도표를 이용해 표시

## 소프트웨어 개발 표준

- ISO/IEC 12207 : ISO에서 만든 표준 소프트웨어 생명 주기 프로세스
- CMMI : 소프트웨어 개발 조직의 업무 능력 및 조직의 성숙도 평가
    - 초기, 관리, 정의, 정량적 관리, 최적화 (기관정량화)
- SPICE : 소프트웨어 품질 및 생산성 향상을 위해 소프트웨어 프로세스를 평가하는 국제 표준
    - 불안전, 수행, 관리, 확립, 예측, 최적화

## 소프트웨어 테일러링

- 소프트웨어 개발 방법론의 절차, 사용기법 등을 수정 및 보완하는 작업
- 법적, 표준 품질 외에는 전부 내부적 기준

## 소프트웨어 개발 프레임워크

소프트웨어 개발에 공통적으로 사용되는 구성 요소와 아키텍처 일반화, 여러가지 기능을 제공해주는 반제품 형태의 소프트웨어 시스템

- 소프트웨어 개발 프레임워크의 특성
    - 모듈화 : 캡슐화하여 모듈화 강화
    - 재사용성 : 재사용이  가능한 모듈을 제공
    - 확장성 : 다형성을 통한 인터페이스 확장이 가능
    - 제어의 역흐름 : 개발자가 관리하고 통제할 객체의 제어를 프레임워크에 넘김

# 2장 : 데이터 입출력 구현

## 데이터베이스 개요

공동으로 사용될 데이터를 중복 배제하여 통합, 저장장치에 저장해 항상 사용할 수 있는 운영 데이터

- DBMS : 사용자 요구에 따라 정보를 생성해주고, 데이터베이스를 관리해주는 소프트웨어
    - 정의, 조작, 제어 기능
- 스키마 : 데이터베이스의 구조와 제약조건에 관한 전반적인 명세를 기술한 것
    - 외부 스키마 : 사용자나 응용 프로그래머가 각 개인의 입장에서 필요로 하는 데이터베이스의 논리적 구조 정의
    - 개념 스키마 : 데이터베이스 전체적인 논리적 구조, 모든 응용 프로그램이나 사용자들이 필요로하는 데이터를 종합한 조직 전체의 데이터베이스, 하나만 존재
    - 내부 스키마 : 물리적 저장장치의 입장에서 본 데이터베이스 구조, 실제 저장될 레코드 형식이나 표현 방법, 물리적 순서 등을 나타냄

## 데이터베이스 설계 순서

1. 요구조건 분석 : 요구 조건 명세서 작성
2. 개념적 설계 : 개념 스키마, 트랜잭션 모델링, E-R 모델
3. 논리적 설계 : 논리 스키마 설계, 트랜잭션 인터페이스 설계
4. 물리적 설계 : 물리적 구조의 데이터로 변환
5. 구현 : DDL로 데이터베이스 생성, 트랜잭션 작성
- 개념적 설계 : 현실 세계에 대한 인식을 추상적 개념으로 표현하는 과정
- 논리적 설계 : 현실 세계에서 발생하는 자료를 특정 DBMS가 지원하는 논리적 자료 구조로 변환시키는 과정
- 물리적 설계 : 논리적 구조로 표현된 데이터를 디스크에 저장하게끔 물리적 구조의 데이터로 변환

## 데이터 모델

데이터 모델은 현실 세계의 정보들을 컴퓨터에 표현하기 위해 체계적으로 표현한 개념적 모형이다.

- 데이터 모델에 표시할 요소
    - 구조 (Structure) : 논리적으로 표현된 개체 타입들 간의 관계로서, 데이터 구조 및 정적 성질 표현
    - 연산 (Operation) : 데이터베이스에 저장된 실제 데이터를 처리하는 작업에 대한 병세로서, 데이터베이스를 조작하는 기본 도구
    - 제약 조건 (Constraint) : 데이터베이스에 저장될 수 있는 실제 데이터의 논리적인 제약 조건

## E-R (개체-관계) 모델

개체와 개체간의 관계를 기본 요소로 이용하여 현실 세계의 무질서한 데이터를 개념적인 논리 데이터로 표현하기 위한 방법

- 사각형 : 개체 타입
- 마름모 : 관계 타입
- 타원 : 속성
- 이중 타원 : 다중 속성
- 선, 링크 : 개체 타입과 속성을 연결

## RDB의 데이터베이스 구조

관계형 데이터베이스는 2차원적인 표를 이용해 데이터 상호 관계를 정의하는 데이터베이스

- 릴레이션의 구조
    - 튜플(tuple, record, row) : 릴레이션을 구성하는 각각의 행
        - 튜플의 총 수는 카디널리티 혹은 기수
    - 속성(attribute, field, column) : 데이터베이스를 구성하는 가장 작은 논리적 단위
        - 속성의 총 수는 디그리 또는 차수
    - 도메인 : 하나의 속성이 가질 수 있는 같은 타입의 원자 값들의 집합
    - 릴레이션 스키마 + 릴레이션 인스턴스 = 릴레이션
- 릴레이션의 특징
    - 튜플 사이에는 순서가 없으며, 삽입, 삭제 등으로 인해 시간에 따라 변한다.
    - 속성들 간의 순서는 중요하지 않으며, 속성 값은 더 이상 쪼갤 수 없는 원자값만을 저장한다.

## Key

- 슈퍼키 (Super Key)
    - 릴레이션 내 속성들의 집합으로 구성된 키
    - 유일성은 만족하지만, 최소성은 만족하지 않음
- 후보키 (Candidate Key)
    - 속성들 중 튜플을 유일하게 식별하기 위해 사용되는 속성들의 부분집합
    - 유일성과 최소성을 모두 만족함
        - 유일성 : 하나의 키 값으로 하나의 튜플만을 유일하게 식별할 수 있어야 함
        - 최소성 : 키를 구성하는 속성 하나를 제거하면 유일하게 식별할 수 없도록, 최소의 속성으로 구성되어야 함
- 기본키 (Primary Key)
    - 후보키들 중 특별히 선정된 주키
    - NULL값이 들어가지 않으며, 중복되지 않는다.
- 대체키 (Alternate Key)
    - 후보키가 둘 이상일 때, 기본키를 제외한 나머지 후보키
- 외래키 (Foreign Key)
    - 다른 릴레이션의 기본키를 참조하는 속성, 또는 속성들의 집합을 의미
    - 외래키로 지정되면 참조 릴레이션의 기본키에 없는 값은 입력 불가

## 무결성 (Integrity)

무결성은 데이터베이스에 저장된 데이터 값과 그것이 표현하는 현실세계의 값이 일치하는 정확성을 의미한다.

신뢰성을 보장하기 위해, 의도하지 않은 요인에 의해 데이터나 시스템이 변경되거나 손상되지 않고 완전성, 정확성, 일관성을 유지함을 보장하는 특성

- 개체 무결성 : 기본키는 NULL값을 가질 수 없으며, 중복값을 가질 수 없다.
- 참조 무결성 : 외래키는 NULL이거나, 참조 릴레이션의 기본키 값과 동일해야 한다.
- 도메인 무결성 : 주어진 속성 값이 정의된 도메인에 속한 값이어야 한다.
- 사용자 정의 무결성 : 속성 값들이 사용자가 정의한 제약조건에 만족되어야 한다.

## 순수 관계 연산자

- Select : σ
    - 릴레이션에 존재하는 튜플 중에서 선택 조건을 만족하는 튜플의 부분 집합을 구하여 새로운 릴레이션을 만드는 연산
- Project : π
    - 주어진 릴레이션에서 속성 리스트(Attribute List)에 제시된 속성 값만을 추출하여 새로운 릴레이션을 만드는 연산
- Join : ▷, ◁
    - 공통 속성을 중심으로 두 개의 릴레이션을 하나로 합쳐서 새로운 릴레이션을 만드는 연산자
- Division : ÷
    - 릴레이션 A, B가 있을 때 릴레이션 B의 조건에 맞는 것들만 릴레이션 A에서 분리하여 프로젝션을 하는 연산

## 일반 집합 연산자

수학적 집합 이론에서 사용하는 연산자이다.

- UNION (합집합) : 두 릴레이션에 존재하는 튜플의 합집합, 중복된 튜플은 제거되는 연산
- INTERSECTION (교집합) : 두 릴레이션에 존재하는 튜플의 교집합을 구하는 연산
- DIFFERENCE (차집합) : 두 릴레이션에 존재하는 튜플의 차집합을 구하는 연산
- CARTESIAN PRODUCT (교차곱) : 두 릴레이션에 있는 튜플들의 순서쌍을 구하는 연산
    - 교차곱의 디그리는 두 릴레이션의 디그리를 더한 것과 같고, 카디널리티는 두 릴레이션의 카디널리티를 곱한 것과 같다.

## 관계해석

관계 해석은 관계 데이터의 연산을 표현하는 방법이다.

- 원하는 정보가 무엇이라는 것만 정의하는 비절차적 특성
- 관계 대수는 정보를 검색하기 위한 절차적인 언어

## 이상 (Anomaly)

테이블에서 일부 속성들의 종속으로 인해 데이터의 중복이 발생하고, 이 중복으로 인해 테이블 조작 시 문제가 발생하는 현상

- 삽입 이상 : 테이블에 데이터 삽입 시 의도와 상관 없이 원하지 않는 값들로 인해 삽입이 불가능
- 삭제 이상 : 한 튜플을 삭제할 때 의도와는 상관 없는 값들도 함께 삭제되는 연쇄 삭제 발생
- 갱신 이상 : 튜플에 있는 속성 값을 갱신할 때 일부 튜플의 정보만 갱신돼 정보의 불일치성이 발생

## 함수적 종속

시간에 관계 없이 X의 값 각각에 대응하는 Y값이 오직 하나라면, Y는 X에 함수적 종속되며 X→Y

- X→Y에서 X를 결정자, Y를 종속자라고 한다.
- 학번 → 이름일 경우 학번만 있다면 이름이 결정되며, 학번이 결정자, 이름이 종속자이다.
- 완전 함수적 종속 : X→Y이면서 X의 진부분 집합엔 함수적 종속이 아닐 때
- 부분 함수적 종속 : X→Y이면서 X의 진부분 집합에 함수적 종속일 때
    - 예를 들어 (학번, 과목번호)가 기본키일 때, 성적은 학번과 과목번호 모두가 있어야 하나 구해지므로 완전 함수적 종속. 그러나 학년은 과목번호는 필요 없고 학번만 있어도 하나 구해지므로 부분 함수적 종속이다.
- 이행적 함수적 종속 : X→Y이고 Y→Z일 때 X→Z를 만족하는 관계

## 정규화

테이블의 속성들이 상호 종속적인 관계를 갖는 특성을 이용해 테이블을 무손실 분해하는 과정

- 제 1정규형 : 모든 속성의 도메인이 원자값으로만 되어있는 정규형
- 제 2정규형 : 기본키가 아닌 모든 속성이 기본키에 대해 완전 함수적 종속을 만족하는 정규형
    - 즉, 부분 함수적 종속을 제거하는 것이 제 2정규화
- 제 3정규형 : 기본키가 아닌 모든 속성이 기본키에 대해 이행적 함수적 종속을 만족하지 않는 정규형
- BCNF : 모든 결정자가 후보키인 정규형
- 제 4정규형 : 다중 값 종속 (다치 종속) 제거
- 제 5정규형 : 조인 종속성을 이용해 후보키를 통해서만 성립되는 과정
- 정규화 과정 : 도메인 원자값 + 부이결다조

## 반정규화 (비정규화)

시스템 성능 향상 및 편의성 증가를 위해 정규화된 데이터 모델을 의도적으로 통합, 중복, 분리해 정규화 원칙을 위배하는 행위

반정규화 수행 시 성능 향상, 관리 효율이 증가하나 일관성 및 정합성이 저하될 수 있다

- 테이블 통합
- 테이블 분할
    - 수평 분할은 레코드를 자르는 것, 수직 분할은 속성이 많아 속성을 자르는 것
- 중복 테이블 추가 : 작업 효율을 향상시키기 위해 중복된 테이블을 추가
- 중복 속성 추가 : 데이터 조회 경로를 단축하기 위해 자주 사용하는 속성 하나 더 추가

## 시스템 카탈로그

시스템 그 자체에 관련이 있는 다양한 객체에 관한 정보를 포함하는 시스템 데이터베이스이다.

- DBMS가 스스로 생성하고 유지하며, 데이터 사전이라고도 불린다.
- 시스템 카탈로그엔 메타 데이터가 저장되어 있다.

## 트랜잭션

데이터베이스의 상태를 변환시키는 하나의 논리적 기능을 수행하기 위한 작업의 단위 또는 한꺼번에 모두 수행되어야 할 일련의 연산들을 의미한다.

### 트랜잭션의 특성 (ACID)

- 원자성 (Atomicity) : 트랜잭션은 모두 반영되거나 전혀 반영되지 않아야 한다 (커밋, 롤백)
- 일관성 (Consistency) : 트랜잭션 완료 후 언제나 일관성있는 데이터베이스 상태여야 한다
- 독립성 (Isolation) : 어느 하나의 트랜잭션 실행 도중 다른 트랜잭션이 끼어들 수 없다
- 지속성 (Durability) : 성공적으로 완료된 트랜잭션은 고장나도 영구적으로 반영되어야 한다

## CRUD 분석

프로세스와 테이블 간에 CRUD 매트릭스를 만들어 트랜잭션을 분석하는 것

- 이를 통해 트랜잭션 분석을 할 수 있다.

## 뷰

사용자에게 접근이 허용된 자료만 제한적으로 보여주기 위해 하나 이상의 기본 테이블로부터 유도된 이름을 가지는 가상 테이블

- 저장장치 내에 물리적으로 존재하지 않음, 있는 것처럼 간주됨
- 독립적인 인덱스를 가질 수 없으며, 정의를 변경할 수 없고, 뷰로 구성된 내용엔 삽입, 삭제, 갱신 연산에 제약이 따름

## 파티션

DB에서 파티션은 대용량의 테이블이나 인덱스를 작은 논리적 단위인 파티션으로 나누는 것

- 범위 분할 (Range) : 지정한 열의 값을 기준으로 분할 (일, 월, 분기별 등)
- 해시 분할 (Hash) : 해시 함수를 적용한 결과 값에 따라 데이터를 분할
- 조합 분할 (Composite) : 범위 분할로 분할 후 해시 함수를 적용해 분할

## RTO/RPO

- RTO : 비상사태 또는 업무 중단시점으로부터 복구 및 가동까지의 소요 시간 (Time)
- RPO : 비상사태 또는 업무 중단시점으로부터 복구 가능한 기준점 (Point)

## 접근 통제

데이터가 저장된 객체와 이를 사용하려는 주체 사이의 정보 흐름을 제한하는 것

- 접근 통제 3요소로는 정책 / 메커니즘 / 보안모델이 있다.
- 접근 통제 기술
    - 임의 접근통제(DAC) : 데이터에 접근하는 사용자의 신원에 따라 접근 권한을 부여
        - 데이터 소유자가 권한을 지정하고 제어
    - 강제 접근통제(MAC) : 주체와 객체의 등급을 비교해 접근 권한을 부여하는 방식
        - 시스템이 접근통제 권한을 지정
    - 역할기반 접근통제(RBAC) : 사용자의 역할에 따라 접근 권한 부여
        - 중앙관리자가 접근 통제 권한 지정

## 데이터베이스 백업

전산 장비의 장애에 대비해 저장된 데이터를 보호하고 복구하기 위한 작업

- 로그 파일 : 데이터베이스의 처리 내용이나 이용 상황 등 상태 변화를 시간의 흐름에 따라 모두 기록한 파일
    - UNDO : 트랜잭션들이 작업한 내용을 모두 취소한다.
    - REDO : 로그를 이용해 해당 데이터 항목에 대해 이전 값을 이후 값으로 재작업한다.

## 스토리지

대용량 데이터를 저장하기 위해 서버와 저장장치를 연결하는 기술

- DAS : 서버와 저장장치를 전용 케이블로 직접 연결
- NAS : 서버와 저장장치를 네트워크를 통해 연결
- SAN : 서버와 저장장치를 연결하는 전용 네트워크를 별도로 구성, DAS와 NAS 혼합

## 자료구조

- 선형구조 : 배열, 선형 리스트, 스택, 큐, 데크
- 비선형구조 : 트리, 그래프
    - 방향 그래프의 최대 간선 수 : n(n - 1)
    - 무방향 그래프에서 최대 간선 수 : n(n-1)/2

## 트리

정점과 선분을 이용해 사이클을 이루지 않도록 구성한 그래프의 특수한 형태

- 근노드 : 맨 위에 있는 노드
- 디그리(차수) : 각 노드에서 뻗어나온 가지의 수
- 트리의 디그리 : 최대 디그리
- 단말 노드 : 자식이 하나도 없는 노드
- 깊이 : 해당 트리의 최대 레벨, 최대 깊이

## 이진트리

- 전위 운행 (Preorder) : 루트 → 왼쪽 → 오른쪽
- 중위 운행 (Inorder) : 왼쪽 → 루트 → 오른쪽
- 후위 운행 (Postorder) : 왼쪽 → 오른쪽 → 루트
- 수식의 경우 루트 대신 연산자를 넣고, 왼쪽 오른쪽 대신 AB를 넣으면 치환된다.
- 중위 표기를 전위, 후위로 바꿀 땐 연산자를 왼쪽 / 오른쪽으로 이동시키면 됨
- 전위, 후위 표기를 중위로 바꿀 땐 연산자, A, B 한 세트를 묶고 연산자를 가운데에 위치시키기

## 정렬

- 삽입정렬 : 가장 간단한 정렬 방식, 2번 파일부터 마지막 파일까지 앞에서부터 자신이 있을 곳으로 이동
- 선택정렬 : 1번 파일과 나머지를 싹다 비교해서 최솟값 앞에 두고, 2번 파일과 나머지를 싹다 비교해서 최솟값 앞에 두고를 반복
- 버블정렬 : 팔이 가장 짧은 방식, 가장 뒤에 최댓값이 먼저 쌓이며 하나씩 톡톡 교체
    - 위 3개는 모두 시간 복잡도 O(n^2)
- 퀵 정렬 : 키를 기준으로 작은 값은 왼쪽, 큰 값은 오른쪽 서브 파일에 분해시키는 방식
    - 평균 O(nlog2n) 최악 O(n^2)
- 힙 정렬 : 전이진 트리를 이용한 정렬 방식
- 2-Way 합병 정렬 : 이미 정렬된 두 파일 한개로 합치기
    - 위 2개는 모두 시간 복잡도 O(nlog2n)

## 소프트웨어 아키텍처

소프트웨어를 구성하는 요소들 간의 관계를 표현하는 시스템의 구조 또는 구조체

- 모듈화 : 성능향상, 재사용, 유지보수가 용이하도록 시스템 기능들을 모듈 단위로 나눈 것
- 추상화 : 문제의 전체적이고 포괄적인 개념을 설계 후 구체화시켜 나가는 것
- 단계적 분해 : 문제를 상위 중요 개념부터 하위 개념으로 구체화시키는 분할 기법
- 정보 은닉 : 한 모듈 내부에 포함된 절차와 자료들의 정보가 감추어져 다른 모듈이 접근하거나 변경하지 못하도록 하는 기법
- 상위 설계 : 아키텍처 설계, 예비 설계, 전체적인 구조, DB, 인터페이스
- 하위 설계 : 모듈 설계, 상세 설계, 내부 구조 및 행위, 자료구조, 알고리즘

## 아키텍처 패턴

아키텍처를 설계할 때 참조할 수 있는 전형적인 해결 방식 또는 예제

- 파이프-필터 패턴 : 데이터 스트림 절차의 각 단계를 필터로 캡슐화, 파이프를 통해 전송
- MVC 패턴 : 서브시스템을 모델, 뷰, 컨트롤러로 구조화하는 패턴
- 마스터-슬레이브 패턴 : 슬레이브 컴포넌트에서 처리된 결과물을 다시 돌려받는 방식

## 객체지향

소프트웨어의 각 요소들을 객체로 만든 후 조립해서 소프트웨어를 개발하는 기법

- 객체(Object) : 데이터와 이를 처리하기 위한 함수를 묶어놓은 모듈
- 클래스 : 공통된 속성과 연산을 갖는 객체의 집합
- 메시지 : 객체들 간의 상호작용을 하는데 사용되는 수단
- 캡슐화 : 외부에서 접근을 제한하기 위해 인터페이스를 제외한 세부 내용을 은닉하는 것
- 다형성 : 하나의 메시지에 대해 각각의 객체가 고유의 방법으로 응답할 수 있는 능력
    - 오버라이딩, 오버로딩
- 연관성 : 두 개 이상의 객체들이 상호 참조하는 관계
    - is (a) part of : 집단화, 관련있는 객체들을 묶어 하나의 상위 객체를 구성
    - is a : 일반화, 공통적인 성질들로 추상화한 상위 객체를 구성

## 객체지향 분석 및 설계

객체지향 분석(OOA)은 사용자의 요구사항과 관련된 객체, 속성, 연산, 관계 등을 정의하여 모델링하는 작업이다.

- 럼바우 방법 : 분석 활동을 객체, 동적, 기능 모델로 나누어 수행 (객동기)
- 부치 방법 : 미시적(Micro) 개발과 거시적(Macro) 개발을 모두 사용
- 자콥슨 방법 : 유스케이스를 강조해 사용
- Coad와 Yourdon 방법 : E-R 다이어그램을 사용해 객체의 행위를 모델링

## 럼바우의 분석 기법

모든 소프트웨어의 구성 요소를 그래픽 표기법을 이용하여 모델링

- 객체 모델링 : 시스템에서 요구되는 객체를 찾아 객체 다이어그램으로 표시
    - Object지만, 정보 모델링 (Information)이라고도 한다.
- 동적 모델링 : 상태 다이어그램을 이용해 시간의 흐름에 따른 객체간 제어 동적 모델링
- 기능 모델링 : 자료 흐름도(DFD)를 이용해 다수 프로세스들간 자료 흐름 중심 처리 과정

## 객체지향 설계 원칙 (SOLID)

- 단일 책임 원칙(SRP) : 객체는 단 하나의 책임만 가져야 한다
- 개방-폐쇄 원칙(OCP) : 코드를 변경하지 않고 기능을 추가할 수 있어야 한다
- 리스코프 치환 원칙(LSP) : 자식 클래스는 최소한 부모 클래스의 기능을 수행할 수 있다
- 인터페이스 분리 원칙(ISP) : 자신이 사용하지 않는 인터페이스와 의존하지 않아야 한다
- 의존 역전 원칙(DIP) : 의존 관계 성립 시 추상성 높은 클래스랑 의존 관계를 맺어야 한다

## 모듈

모듈화를 통해 분리된 시스템의 각 기능이다.

- 모듈의 독립성은 결합도(Coupling)와 응집도(Cohesion)에 의해 측정된다.

## 결합도

결합도는 모듈 간에 상호 의존하는 정도를 나타낸다. 약할 수록 품질이 높다.

- 자료 결합도 : 모듈 간의 인터페이스가 자료 요소로만 구성될 때의 결합도 (가장 약함)
- 스탬프 결합도 : 모듈 간의 인터페이스로 배열이나 레코드 등의 자료구조가 전달
- 제어 결합도 : 다른 모듈 내부의 논리적 흐름을 제어하기 위해 제어 신호나 요소를 전달
- 외부 결합도 : 어떤 모듈에서의 데이터를 외부의 다른 모듈에서 참조할 때의 결합도
- 공통 결합도 : 공유되는 공통 데이터 영역을 여러 모듈이 사용할 때의 결합도
- 내용 결합도 : 내부 기능및 그 내부 자료를 직접 참조하는 결합도 (가장 강함)

## 응집도

모듈의 내부 요소들이 서로 관련되어 있는 정도이다. 강할 수록 품질이 높다.

- 우연적 응집도 : 모듈 내부 각 구성 요소가 서로 관련 없는 요소로만 구성 (가장 약함)
- 논리적 응집도 : 유사한 성격을 갖고나 특정 형태로 분류되는 요소들로 구성
- 시간적 응집도 : 특정 시간에 처리되는 몇 개의 기능을 모아 하나의 모듈로 구성
- 절차적 응집도 : 다수의 관련 기능을 가질 때 모듈 구성 요소가 그 기능을 순차적으로 수행
- 교환적 응집도 : 동일한 입력과 출력을 사용해 서로 다른 기능을 수행하는 구성 요소들
- 순차적 응집도 : 모듈 내 하나의 활동으로부터 나온 출력 데이터를 다음 활동의 입력으로 사용
- 기능적 응집도 : 모든 기능 요소들이 단일 문제와 연관되어 수행될 경우 (가장 강함)

팬아웃은 어떤 모듈에 의해 제어되는 모듈의 수

## 디자인 패턴

모듈 간의 관계 및 인터페이스를 설계할 때 참조할 수 있는 전형적인 해결 방식 또는 예제

### 생성 패턴

- 추상 팩토리(Abstract Factory) : 구체적인 클래스 의존X, 인터페이스를 통해 서로 연관하는 객체들을 그룹으로 생성해 추상적으로 표현
- 빌더 (Bulider) : 작게 분리된 인스턴스를 건축하듯 조합해 객체를 생성
- 팩토리 메소드 (Factory Method) : 상위 클래스에서 인터페이스만 정의, 서브 클래스에서 객체를 생성
- 프로토타입 (Prototype) : 원본 객체를 복제하는 방법으로 객체 생성
- 싱글톤 (Singleton) : 하나의 객체 생성 시 어디서든 참조, 여러 프로세스 동시 참조 안됨

### 구조 패턴

- 어댑터 (Adapter) : 호환성 없는 클래스들의 인터페이스를 다른 클래스가 이용하도록 변환
- 브릿지 (Bridge) : 구현부에서 추상층 분리, 서로가 독립적으로 확장할 수 있도록 구상
- 프록시 (Proxy) : 접근이 어려운 객체에 연결하려는 객체 사이에서 인터페이스 역할을 수행
- 컴포지트, 데코레이터, 퍼싸드, 플라이웨이트

### 행위 패턴

- 책임 연쇄 (Chain of Responsibility) : 하나의 클래스에서 처리가 안되면 다음 클래스가 처리하게
- 커맨드 (Command) : 요청을 객체의 형태로 캡슐화, 재이용하거나 취소할 수 있게 정보 저장, 로그
- 중재자 (Mediator) : 수많은 객체들 간 복잡한 상호작용을 캡슐화해 객체로 정의
- 옵저버 (Observer) : 한 객체의 상태 변화 시 상속된 다른 객체들에게 변화된 상태를 전달
- 상태 (State) : 객체의 상태에 따라 동일한 동작을 다르게 처리할 때 사용
- 전략 (Strategy) : 동일한 개념의 알고리즘들을 개별적으로 캡슐화해 상호 교환할 수 있게 정의
- 템플릿 메서드 (Template Method) : 상위 클래스에서 골격, 하위 클래스에서 세부 처리를 구체화
- 방문자 (Visitor) : 각 클래스들의 데이터 구조에서 처리 기능을 분리하여 별도의 클래스로 구성

# 7장 : 애플리케이션 테스트 관리

## 애플리케이션 테스트

애플리케이션에 잠재되어 있는 결함을 찾아내는 일련의 행위 또는 절차

- 애플리케이션 테스트의 기본 원리
    - 완벽한 테스트 불가능
    - 파레토 법칙 : 20% 코드에서 전체 결함의 80%가 발견
    - 살충제 패러독스 : 동일한 테스트 케이스 반복하면 더 이상 결함이 발견되지 않음
    - 오류-부재의 궤변 : 소프트웨어 결함을 모두 제거해도 요구사항 만족 못하면 별로임

## 애플리케이션 테스트의 분류

- 실행 여부에 따른 테스트 분류
    - 정적 테스트 : 워크스루, 인스펙션, 코드 검사 등 실행하지 않고 명세서나 소스코드 분석
    - 동적 테스트 : 프로그램을 실행하여 오류 찾기 (블랙박스, 화이트박스)
- 시각에 따른 테스트
    - 검증 테스트 : 개발자의 관점에서 생산 과정 테스트
    - 확인 테스트 : 사용자의 관점에서 제품의 결과 테스트
- 목적에 따른 테스트
    - 회복 테스트 : 여러 결함을 주고 스스로 복구되는지 확인
    - 안전 테스트 : 불법적인 침입으로부터 시스템 보호하는지 확인
    - 강도 테스트 : 과도한 정보량이나 빈도를 부과해 과부하시에도 실행되는지 확인

## 화이트박스 테스트

원시 코드의 논리적인 모든 경로를 테스트하여 테스트 케이스를 설계하는 방법

- 테스트의 종류
    - 기초 경로 검사 : 절차적 설계의 논리적 복잡성을 측정할 수 있는 테스트
    - 제어 구조 검사 : 조건 검사, 루프 검사, 데이터 흐름 검사
- 테스트의 검증 기준
    - 문장 검증 기준 : 소스 코드의 모든 구문이 한 번 이상 수행되도록
    - 분기 검증 기준 : 모든 조건문에 대해 참/거짓 한 번 이상 수행되도록
    - 조건 검증 기준 : 조건문이 포함된 개별 조건식 결과가 참/거짓 한 번 이상 수행되도록
    - 분기/조건 기준 : 분기 검증과 조건 검증 모두 만족하는 설계

## 블랙박스 테스트

각 기능이 완전히 작동되는 것을 입증하는 테스트

- 테스트의 종류
    - 동치 분할 검사 : 프로그램의 입력 조건에 타당한 입력 자료, 타장하지 않은 입력 자료의 개수를 균등하게 하여 테스트 케이스를 정하고, 입력 자료에 맞는 결과가 출력되는지 확인
    - 경계값 분석 : 입력 조건의 중간값보다 경계값에서 오류가 발생될 확률 높음. 입력 조건의 경계값을 테스트 케이스로 선정
    - 원인-효과 그래프 검사 : 입력 데이터 간의 관계와 출력에 영향을 미치는 상황을 체계적으로 분석, 효용성이 높은 테스트 케이스를 선정하여 검사
    - 오류 예측 검사, 비교 검사

## 개발 단계에 따른 애플리케이션 테스트

- 소프트웨어 개발 단계에 따라 단위 - 통합 - 시스템 - 인수 테스트로 분류된다.
- 그 앞에 소프트웨어 개발 단계인 요구사항 - 분석 - 설계 - 구현을 붙인것이 V-모델이다.
- 단위 테스트 : 모듈이나 컴포넌트에 초점을 맞춰 테스트하는 것
- 통합 테스트 : 단위 테스트가 완료된 모듈을 결합, 하나의 시스템으로 만드는 과정에서 테스트
- 시스템 테스트 : 개발된 소프트웨어가 완벽하게 수행하는가를 점검하는 테스트
- 인수 테스트 : 사용자의 요구사항을 충족하는지에 중점을 두고 테스트
    - 알파 테스트 : 개발자 장소에서 통제된 환경에서 행해지는 테스트
    - 베타 테스트 : 선정된 최종 사용자가 직접 실업무로 테스트

## 통합 테스트

- 비점진적 통합 방식 : 단계적인 절차 없이 모든 모듈이 미리 결합된 상태로 전체 테스트
    - 빅뱅 통합 테스트
- 점진적 통합 방식 : 모듈 단위로 단계적으로 통합하는 방식

### 하향식 통합 테스트

상위 모듈에서 하위 모듈 방향으로 통합하면서 테스트

- 주요 제어 모듈은 작성된 프로그램 사용, 종속 모듈은 스텁으로 대체한다.
- 스텁 : 제어 모듈이 호출하는 타 모듈의 기능을 단순히 수행하는 도구, 일회용 모듈. 하향식.

### 상향식 통합 테스트

하위 모듈에서 상위 모듈 방향으로 통합하면서 테스트

- 하위 모듈을 클러스터로 결합한다.
- 상위 모듈에서 데이터의 입출력을 확인하기 위해 더미 모듈인 드라이버를 작성한다.
- 드라이버 : 테스트 대상의 하위 모둘 호츌, 파라미터 전달, 모듈 테스트 결과 도출 도구.

### 혼합식 통합 테스트

하위 수준에선 상향식 통합, 상위 수준에서는 하향식 통합을 사용해 최적의 테스트를 지원하는 방식

### 회귀 테스팅

통합 테스트로 인해 변경된 모듈이나 컴포넌트에 새로운 오류가 있는지 확인하는 테스트

- 테스트를 반복하며 새로운 오류가 발생하지 않음을 보증하기 위한 반복 테스트

## 테스트 케이스

구현된 소프트웨어가 사용자의 요구사항을 정확히 준수했는지 확인하기 위한 테스트 항목 명세서

- 테스트 시나리오 : 테스트 케이스를 적용하는 순서에 따라 여러 개의 테스트 케이스를 묶은 집합

## 테스트 오라클

테스트 결과가 올바른지 판단하기 위해 사전에 정의된 참값을 대입해 비교하는 기법

- 참 오라클 : 모든 테스트 케이스의 입력 값에 대해 기대하는 결과 제공 오라클
- 샘플링 오라클 : 특정한 몇몇 테스트 케이스 입력 값에 대해서만 기대하는 결과 제공
- 추정 (Heuristic) 오라클 : 특정 테스트 케이스 입력 값에 대해 결과 제공, 나머지는 추정 제공
- 일관성 검사 (Consistent) 오라클 : 애플리케이션 변경 후 수행 전과 후의 테스트 결과 동일한지

## 테스트 하네스 도구

테스트가 실행될 환경을 시뮬레이션하여 컴포넌트 및 모듈이 정상적으로 테스트되도록 하는 도구

테스트 하네스 : 애플리케이션 테스트 환경 일부분, 테스트 지원용 생성된 코드와 데이터

- 테스트 드라이버 : 하위 모듈을 호출, 파라미터 전달, 결과 도출
- 테스트 스텁 : 제어 모듈이 호출하는 타 모듈의 기능을 단순히 수행하는 도구, 일시적 조건 모듈
- 테스트 슈트 : 테스트 대상의 컴포넌트나 모듈에 사용되는 테스트 케이스의 집합

## 결함

- 자원 사용률 : 의뢰한 작업을 처리하는 동안의 CPU, 메모리, 네트워크 사용량 등

## 시간 복잡도

알고리즘을 수행하기 위해 프로세스가 수행하는 연산 횟수를 수치화한 것

- 빅오 표기법
    - O(1) : 입력값 관계 없이 일정하게 문제 해결에 하나의 단계만 거침
    - O(log2n) : 입력값 또는 조건에 의해 필요 단계가 감소
    - O(n) : 문제 해결에 필요한 단계가 입력값과 1:1 관계
    - O(nlog2n) : 문제 해결에 필요한 단계가 n(log2n)번 수행
        - 힙 정렬, 2-way 합병 정렬
- 순환 복잡도 : 논리적인 복잡도를 측정하기 위한 소프트웨어의 척도
    - 화살표의 수 - 노드의 수 + 2

# 8장 : SQL 응용

## DDL

DDL은 DB 구조, 데이터 형식, 접근 방식 등 DB를 구축하거나 수정할 목적으로 사용하는 언어이다.

- CREATE : 스키마, 도메인, 테이블, 뷰, 인덱스를 정의
- ALTER : 테이블에 대한 정의를 변경
- DROP : 스키마, 도메인, 테이블, 뷰, 인덱스를 삭제

```sql
CREATE TABLE 테이블명 (
	속성명 데이터타입 DEFAULT NOT NULL,
	PRIMARY KEY (기본키 속성명),
	UNIQUE (대체키 속성명),
	FOREIGN KEY (외래키 속성명),
		REFERENCES 참조테이블(기본키 속성명),
		ON DELETE 옵션 // SET NULL
		ON UPDATE 옵션 // CASCADE
	CONSTRAINT 제약조건명
);
```

```sql
ALTER TABLE 테이블명 ADD 속성명 데이터타입 DEFAULT '기본값';
ALTER TABLE 테이블명 ALTER 속성명 SET DEFAULT '기본값'; // 특정 속성 디폴트값 변경
ALTER TABLE 테이블명 DROP COLUMN 속성명 CASCADE; // 특정 속성 삭제
```

## DCL

DCL은 데이터 보안, 무결성, 회복, 병행 제어 등을 정의하는 데 사용하는 언어이다.

- COMMIT : 수행된 결과를 실제 물리 디스크에 저장하고, 정상적으로 완료되었음을 알려줌
- ROLLBACK : 데이터베이스 조작 작업이 비정상적으로 종료되었을 때, 원래 상태로 복구함
    - 변경되었으나 아직 COMMIT되지 않은 모든 내용들을 취소하고, 데이터베이스를 이전 상태로 되돌리는 명령어
- GRANT :  데이터베이스 사용자에게 사용 권한을 부여
- REVOKE : 데이터베이스 사용자의 사용 권한을 취소

```sql
// 사용자 등급 지정 및 해제
GRANT 사용자등급 TO 사용자_ID_리스트;
REVOKE 사용자등급 FROM 사용자_ID_리스트;

// 테이블 및 속성에 대한 권한 부여 및 취소
GRANT 권한_리스트 ON 개체 TO 사용자 [WITH GRANT OPTION];
REVOKE [GRANT OPTION FOR] 권한_리스트 ON 개체 FROM 사용자 [CASCADE];
```

- SAVEPOINT : 트랜잭션 내 ROLLBACK할 위치인 저장점을 지정하는 명령어

```sql
SAVEPOINT S1;

ROLLBACK TO S1;
```

## DML

DML은 데이터베이스 사용자가 저장된 데이터를 실질적으로 관리하는데 사용되는 언어이다.

- SELECT : 테이블에서 튜플을 검색
- INSERT : 테이블에 새로운 튜플 삽입
- DELETE : 테이블에서 튜플 삭제
- UPDATE : 테이블에서 튜플의 내용을 갱신

```sql
INSERT INTO 테이블명(속성명1, 속성명2...) VALUES (데이터1, 데이터2...);
// VALUES 대신 SELECT문으로 원하는 튜플만 검색하는 것도 가능
```

```sql
DELETE

WHERE 조건;
```

```sql

```

## DML - SELECT

```sql
SELECT [PREDICATE] [DISTINCT] [TOP N] 속성명
FROM 테이블명
WHERE 조건
GROUP BY 속성명
HAVING 조건
ORDER BY 속성명 [ASC | DESC];
```

- PREDICATE : 검색할 튜플 수를 제한하는 명령어
- DISTINCT : 중복된 값은 한 번만 출력하는 명령어
- TOP N : 정렬과 같이 쓰며 상위 N개의 튜플만 검색하는 명령어
- AS : 표시할 속성명에 별칭을 둠
- LIKE 연산자

| % | 모든 문자를 대표  |
| --- | --- |
| _ | 문자 하나를 대표 |
| # | 숫자 하나를 대표 |
- NULL 처리를 할 땐 IS NULL 혹은 IS NOT NULL 연산자 사용
- ALL () : ()안에 있는 서브쿼리의 내용을 질의의 조건으로 사용

```sql
SELECT [PREDICATE] [DISTINCT] [TOP N] 속성명
[그룹함수(속성명)]
[WINDOW 함수 OVER(PARTITION BY 속성명)]
FROM 테이블명
WHERE 조건
GROUP BY 속성명
HAVING 조건
ORDER BY 속성명 [ASC | DESC];
```

- 그룹함수 : GROUP BY 절에서 지정된 그룹별로 속성의 값을 집계할 함수
- WINDOW 함수 : GROUP BY절을 이용하지 않고 속성의 값을 집계할 함수
    - PARTITION BY : WINDOW 함수의 적용 범위가 될 속성 지정
- 그룹 함수

| COUNT(속성명) | 그룹별 튜플 수 |
| --- | --- |
| SUM(속성명) | 그룹별 합계 |
| AVG(속성명) | 그룹별 평균 |
| MAX(속성명) | 그룹별 최대값 |
| MIN(속성명) | 그룹별 최솟값 |
| STDDEV(속성명) | 그룹별 표준편차 |
| VARIANCE(속성명) | 그룹별 분산 |
| ROLLUP(속성명, 속성명…) | 그룹별 소계 |
| CUBE(속성명, 속성명,,,) | 모든 조합의 그룹별 소계 |
- WINDOWS함수
    - ROW_NUMBER() : 윈도우별로 각 레코드에 대한 일련번호 반환
    - RANK() : 윈도우별로 순위 반환, 공동 순위 반영
    - DENSE_RANK() : 윈도우별로 순위 반환, 공동 순위 무시

```sql
// <상여금> 테이블에서 '상여내역'별로 '상여금'에 대한 일련번호 구하기
// 단, 순서는 내림차순이며 속성명은 'NO'로 할 것

SELECT 상여내역, 상여금
	ROW_NUMBER() OVER (PARTITION BY 상여내역 ORDER BY 상여금 DESC) AS NO
FROM 상여금;
```

| UNION | 합집합, 중복은 하나만 |
| --- | --- |
| UNION ALL | 합집합, 중복 포함 |
| INTERSECT | 교집합 |
| EXCEPT | 차집합 |

## DML - JOIN

JOIN은 2개의 릴레이션에서 연관된 튜플들을 결합하여, 하나의 새로운 릴레이션을 반환한다.

### INNER JOIN

- INNER JOIN은 EQUI JOIN과 NON-EQUI JOIN으로 구분된다.
    - EQUI JOIN : 공통 속성을 기준으로 ‘=’ 비교를 하는 것
    - NON-EQUI JOIN : ‘=’이 아닌 그 외, >, < 등의 연산자를 쓰는 것
- 조건이 없는 INNER JOIN은 CROSS JOIN과 동일한 결과이다.

```sql
SELECT ~
FROM 테이블명
WHERE 테이블명.속성명 = 테이블명.속성명;

SELECT ~
FROM 테이블명 NATURAL JOIN 테이블명;

SELECT ~
FROM 테이블명 JOIN 테이블명 USING(속성명);
```

### OUTER JOIN

- OUTER JOIN은 JOIN 조건에 만족하지 않는 튜플도 결과로 출력하는 JOIN이다.
- LEFT, RIGHT, FULL OUTER JOIN으로 구분된다.
- 겹치는 조건 문으로 연산자 ON을 사용한다.

```sql
SELECT ~
FROM 테이블명 LEFT OUTER JOIN 테이블명
ON 테이블명.속성명 = 테이블명.속성명;

SELECT ~
FROM 테이블명 RIGHT OUTER JOIN 테이블명
ON 테이블명.속성명 = 테이블명.속성명;

SELECT ~
FROM 테이블명 FULL OUTER JOIN 테이블명
ON 테이블명.속성명 = 테이블명.속성명;
```

## 트리거

트리거는 DB 시스템에서 데이터의 삽입, 갱신, 삭제 등의 이벤트가 발생할 때, 관련 작업이 자동으로 수행되게 하는 절차형 SQL이다.

- 데이터 변경 및 무결성 유지, 로그 메시지 출력 등의 목적으로 사용
- 트리거는 아래 내용을 필수로 포함한다.
    - DECLARE : 트리거의 명칭, 변수 및 상수, 데이터 타입을 정의하는 선언부
    - EVENT : 트리거가 실행되는 조건 명시
    - BEGIN / END : 트리거의 시작과 종료 의미
    - CONTROL : 조건문 또는 반복문 삽입
    - SQL : DML문이 삽입되어 데이터 관리를 위한 조회, 추가, 수정, 삭제 작업 수행
    - EXCEPTION : BEGIN~END까지의 내용 중 예외 발생 시 이를 처리
- 트리거는 DROP TRIGGER 트리거명; 으로 제거한다.

# 9장 : 소프트웨어 개발 보안 구축

## Secure SDLC

보안상 안전한 소프트웨어를 개발하기 위해 SDLC에 보안 강화를 위한 프로세스를 포함한 것

## 소프트웨어 개발 보안 요소

- 기밀성 : 시스템 내 정보와 자원은 인가된 사용자에게만 접근 허용
- 무결성 : 시스템 내 정보는 오직 인가된 사용자만 수정할 수 있음
- 가용성 : 인가받은 사용자는 시스템 내의 정보와 자원을 언제든 사용할 수 있음
- 인증 : 시스템 내 정보와 자원을 사용하려는 사용자가 합법인지 확인하는 행위
- 부인 방지 : 데이터를 송수진한 자가 사실을 부인할 수 없도록 증거를 제공

## 입력 데이터 검증 및 표현

- SQL 삽입 (Injection) : 웹 응용 프로그램에 SQL 삽입, 내부 DB 서버의 데이터 유출 및 변조
- 경로 조작 및 자원 삽입 : 데이터 입출력 경로 조작해 서버 자원 수정, 삭제
- 크로스 사이트 스크립팅(XSS) : 웹페이지에 악의적인 스크립트 삽입, 방문자 정보 탈취
- 운영체제 명령어 삽입 : 외부 입력값을 통해 시스템 명령어 실행, 권한 탈취 및 장애 유발
- 메모리 버퍼 오버플로우 : 연속된 메모리 공간 사용에서 범위 넘어선 위치에서 자료 읽고쓰기

## 보안 기능

- 코드에 직접 비밀번호를 적는 하드코딩된 비밀번호는 브루트포스 어택이나 코드를 보고 뚫릴 위험이 있다.
- 스택 가드 : 널 포인터 역참조와 같이 주소가 저장되는 스택에서 발생하는 보안 약점을 막는 기술. 프로그램의 복귀 주소와 변수 사이에 특정 값 저장 후, 그 값이 변경되면 오버플로우로 판단하여 프로그램 실행을 중단

## 암호 알고리즘

중요한 정보를 보호하기 위해 평문을 암호화된 문장으로 만드는 절차 또는 방법

- 개인키 암호화 기법 : 동일한 키로 데이터를 암호화하고 복호화
    - 대칭 암호 기법 또는 단일키 암호화 기법이라고 한다
    - DES, SEED, AES, ARIA, IDEA, Skipjack
- 공개키 암호화 기법 : 암호화 공개키는 공개, 복호화 비밀키는 비밀리에 관리
    - 비대칭 암호 기법이라고도 한다
    - RSA, ECC

## 보안 솔루션

외부로부터의 불법적인 침입을 막는 기술 및 시스템

- 방화벽 : 내부의 네트워크와 인턴넷 간 전송되는 정보를 선별하여 수용, 거부하는 시스템
- 침입 탐지 시스템(IDS) : 시스템 비정상적인 사용, 오용을 실시간으로 탐지하는 시스템
    - 오용 탐지 : 미리 입력해 둔 공격 패턴 감지되면 알림
    - 이상 탐지 : 평균적인 시스템 상태 기준으로 비정상적인 행위나 사용 감지되면 알림
- 침입 방지 시스템(IPS) : 비정상적인 트래픽을 능동적으로 차단하고 격리하는 보안 솔루션
- VPN : 공중 네트워크와 암호화 기술을 이용해 사용자가 마치 자신의 전용 회선을 사용하는 것처럼 해주는 보안 솔류션
    - SSL 프로토콜을 쓰느냐, IPSec 프로토콜을 사용하느냐에 따라 다른 VPN이다.
- SIEM : 로그 및 보안 이벤트를 통합하여 관리하는 보안 솔루션
- SSH (시큐어 셸) : 다른 컴퓨터에 로그인, 원격 명령 실행 등을 수행할 수 있는 프로토콜
    - 기본적으로 22번 포트를 사용하며, 키를 통한 인증 방법 시 클라잉너트 공개키를 등록
- 템퍼 프루핑 (Tamper Proofing) : 소프트웨어의 위변조가 발생 시 소프트웨어를 오작동하게 만들어 악용을 방지하는 기술
- OAuth (공개 인증) : 인터넷 애플리케이션의 인증 표준 방법으로, 공개 API로 구현

    - 추상 클래스를 정의할 때 쓰는 연산자

실행되는 절차를 중요시하며 데이터를 중심으로 프로시저를 구현, 전체가 유기적으로 연결되어있다.

- ALGOL : 수치 계산이나 논리 연산을 위한 과학 기술 계산용 언어, PASCAL과 C의 모체

- FORTRAN : 과학 기술 계산용 언어, 수학과 공학 분야 공식, 수식을 프로그래밍

## 스크립트 언어

스크립트 언어는 HTML 문서 안에 직접 프로그래밍 언어를 삽입해 사용하는 언어이다.

    - 클라이언트용 : 클라이언트의 웹 브라우저에서 실행됨. Javascript, VBscript
- Javascript : 웹 페이지 동작을 제어하는 언어, 클래스가 존재하지 않으며 변수 선언도 필요 없음

- PHP : 유닉스 계열에서 사용 가능하며, 웹 페이지 제작에 사용됨

- 쉘 스크립트 : 유닉스 계열의 쉘에서 사용되는 명령어들의 조합으로 구성

## 라이브러리

라이브러리는 프로그램을 효율적으로 개발할 수 있도록 자주 사용하는 함수나 데이터들을 미리 만들어 모아놓은 집합체이다.

- 표준 라이브러리 : 언어에 기본적으로 포함되어있는 라이브러리이며, 모듈이나 패키지 형태
- 외부 라이브러리 : 개발자들이 필요한 기능을 만들어 인터넷 등에 공유한 라이브러리, 다운받아 설치함
- C언어의 대표적 표준 라이브러리
    - stdio.h : 데이터의 입출력에 사용되는 기능 제공
    - math.h : 수학 함수를 제공
    - string.h : 문자열 처리에 사용되는 기능 제공
    - stdlib.h : 자료형 변환, 난수, 메모리 동적 할당 기능 제공
    - time.h : 시간 처리에 사용되는 기능 제공
- JAVA의 대표적인 표준 라이브러리
    - 라이브러리를 패키지에 포함해 제공하며, import문을 이용해 선언

## 가상기억장치 구현 기법

가상기억장치는 보조기억장치의 일부를 주기억장치처럼 사용하는 것이다.

- 페이징 기법 : 프로그램과 주기억장치의 영역을 동일한 크기로 나눈 후, 주기억장치의 영역에 적재시켜 실행
    - 외부 단편화는 없으나 내부 단편화가 발생할 수 있다.
- 세그먼테이션 기법 : 프로그램을 다양한 크기의 논리적인 단위로 나눈 후 주기억장치에 적재
    - 내부 단편화는 없으나 외부 단편화가 발생할 수 있다.

## 페이지 교체 알고리즘

페이지 폴트가 발생하면 가상기억장치에서 필요한 페이지를 주기억장치에 적재하는데, 어떤 페이지 프레임을 선택하여 교체할지를 결정하는 알고리즘

- OPT (최적 교체) : 완벽한 알고리즘이며, 앞으로 가장 오랫동안 사용하지 않을 페이지 교체
- FIFO : 가장 먼저 들어와서 가장 오래 있었던 페이지를 교체하는 방법
- LRU : 최근에 가장 오랫동안 사용하지 않은 페이지를 교체하는 기법
    - 먼저 들어온 것과 가장 오랫동안 사용하지 않은것은 확연히 다르다. 이미 메모리에 적재되어있는 페이지가 참조 페이지로 호출될 경우, LRU는 최신에 사용한 것이므로 제거 우선순위가 낮아지지만 FIFO는 그 페이지가 예전부터 있었다면 여전히 제거 우선순우가 높다.
- LFU : 사용 빈도가 가장 적은 페이지를 교체
- NUR : 최근에 가장 사용하지 않은 페이지 교체, 참조 비트와 변형 비트 사용
- SCR : 가장 오랫동안 남은 페이지 중 자주 사용되는 페이지 교체 방지용 2차 기회 교체

## 가상기억장치 기타 관리 사항

- 페이지 크기 : 페이지 크기가 작으면 단편화가 감소되고 페이지 이동 시간이 감소되나 디스크 접근 횟수가 많아져 전체적인 입출력 시간이 늘어나고, 크기가 크면 접근 횟수가 줄고 페이지 맵 테이블 크기가 작아지나 단편화가 증가되고 이동 시간이 늘어남
- Locality (국부성, 지역성) : 프로세스가 실행되는 동안 주기억장치를 참조할 때 일부 페이지만 집중적으로 참조하는 성질
    - 시간 구역성 : 프로세스가 실행되면서 하나의 페이지 주변을 일정 시간동안 집중적으로 접근
    - 공간 구역성 : 프로세스가 실행되면서 일정 위치의 페이지 집중적으로 액세스
- 워킹 셋 : 프로세스가 일정 시간동안 자주 참조하는 페이지들의 집합
    - 자주 참조되는 워킹 셋을 주기억장치에 상주시키면 페이지 폴트가 줄어든다
- 스래싱 : 프로세스의 처리 시간보다 페이지 교체에 소요되는 시간이 더 많아지는 현상
    - 다중 프로그래밍의 정도가 높아져 이용률이 어느 시점까진 늘어나지만 더욱 커지면 스래싱의 정도가 더욱 커진다.

## 프로세스의 개요

프로세스는 실행중인 프로그램을 의미하며, PCB라는 프로세스 정보 태그가 있다.

- 프로세스 상태 전이
    - 제출 (submit) : 작업을 처리하기 위해 사용자가 작업을 시스템에 제출
    - 접수 (hold) : 제출된 작업이 스풀 공간인 디스크의 할당 위치에 저장된 상태
    - 준비, 실행, 대기, 종료
- Dispatch : 준비 상태에서 대기중인 프로세스 하나가 프로세서 할당받고 실행으로 전이
- Wake up : 입출력 작업이 완료되어 대기상태에서 준비 상태로 전이되는 과정
- 스레드 : 시스템의 여러 자원을 할당바당 실행하는 프로그램의 단위
- 비선점 스케줄링 : 이미 할당된 CPU를 다른 프로세스가 강제로 뻇을 수 없는 스케줄링
    - FCFS, SJF, 우선순위, HRN, 기한부
- 선점 스케줄링 : 우선순위가 높은 프로세스가 CPU를 강제로 빼앗을 수 있는 스케줄링
    - RR, SRT, 선점 우선순위, 다단계 큐

## 스케줄링 알고리즘

- FCFS : 준비상태 큐에 도착한 순서대로 CPU 할당 (가장 간단)
- SJF (단기 작업 우선) : 준비 상태 큐에서 실행시간이 가장 짧은 프로세스에게 먼저 CPU를 할당하는 기법, 가장 적은 평균 대기시간
- HRN : 대기 시간과 서비스 시간을 이용하는 방법
    - 실행 시간이 긴 프로세스에 불리한 SJF 기법을 보완
    - 우선순위를 계산하고, 그 숫자가 가장 높은 것부터 우선순위 부여
    - 우선순위 = (대기 시간 + 서비스 시간)/서비스 시간
- RR : 시간 할당량 동안만 실행한 후 다음 프로세스에게 CPU 넘기기
    - 시간 할당이 클 수록 FCFS 기법과 같아지고, 작아질수록 오버헤드가 자주 발생된다.
- SRT : 현재 실행중인 프로세스의 남은 시간과 새로 도착한 프로세스의 실행 시간을 비교하여 가장 짧은 실행 시간을 요구하는 프로세스에게 CPU 할당

## IP 주소

인터넷에 연결된 모든 컴퓨터 자원을 구분하기 위한 고유한 주소

- 8비트씩 4부분으로 총 32빝트로 구성
- 0~127까지가 A클래스, 128~191이 B클래스, 192~223이 C클래스, 224~239 D클래스
- 서브네팅 : 네트워크 주소를 다시 여러 개의 작은 네트워크로 나누어 사용하는 것
    - 192.168.1.0/24에서 /24가 서브넷 마스크, 1의 개수가 24개라는 뜻
    - 즉 1111111.11111111.11111111.00000000 이 서브넷 마스크
    - 0인 부분을 원하는대로 갖고 놀 수 있다. 4개의 서브넷으로 나눌 경우
    - 192.168.1.0 ~ 192.168.1.63 / 192.168.1.64 ~ 192.168.1.127
        - 여기서 첫 번째 주소는 네트워크 주소, 마지막 주소는 브로드캐스트 주소
- IPv6 : IPv4의 주소 부족 문제를 해결하기 위해 개발되었으며 128비트 주소 사용
    - 16비트씩 8부분으로 총 128비트
    - 각 부분을 16진수로 표현하며 콜론(:) 으로 구분한다
    - 유니캐스트(1대1), 멀티캐스트(1대다), 애니캐스트 (가장 가까운 1대1)

## OSI 참조 모델

- 물리계층 : 전송에 필요한 두 장치 간의 실제 접속, 절단, 기계적

- 데이터링크 계층 : 두 개의 인접한 개방 시스템이 신뢰성있는 정보 전송을 할 수 있게 연결 설정, 유지, 흐름 제어, 프레임 동기화, 오류 검출
    - HDLC, LAPB, 랜카드, 브릿지 (LAN과 LAN 연결) , 스위치 (브릿지와 유사하게 LAN 연결)
- 네트워크 계층 : 네트워크 연결을 관리하는 기능, 데이터 교환, 중계, 경로 설정, 트래픽 제어
    - X.25, IP, 라우터 (LAN 연결에 데이터 전송 최적 경로 선택)

    - TCP, UDP, 게이트웨이 (프로토콜 구조가 다른 네트워크 연결)
- 세션 계층 : 송수신 측의 관련성 유지, 대화 제어, 토큰 사용
- 표현 계층 : 응용 계층으로부터 받은 데이터를 통역, 변환, 코드 변환, 암호화, 압축
- 응용 계층 : 사용자가 OSI 환경에 접근하도록 서비스 제공

## 프로토콜

서로 다른 기기들간의 데이터 교환을 원활하게 수행할 수 있도록 표준화시켜놓은 통신 규약

- 구문 (Syntax) : 전송하고자 하는 데이터의 형식, 부호화, 신호 레벨
- 의미 (Semantics) : 두 기기 간의 효율적이고 정확한 정보 전송을 위한 협조 사항과 오류 관리를 위한 제어 정보 규정
- 시간 (Timing) : 두 기기 간의 통신 속도, 메시지 순서 제어 규정

## TCP/IP

서로 다른 기종의 컴퓨터들이 데이터를 주고받을 수 있도록 하는 표준 프로토콜

- TCP : OSI 7계층의 전송 계층에 해당, 가상 회선 방식을 기반으로 하는 연결형 서비스
- IP : OSI 7계층의 네트워크 계층에 해당, 데이터그램 방식 기반의 비연결형 서비스
- TCP/IP의 구조
    - 응용 계층 : 응용 프로그램 간의 데이터 송수신
        - 텔넷, FTP, SMTP, SNMP, DNS, HTTP
    - 전송 계층 : 호스트들 간의 신뢰성 있는 통신 제공
        - TCP, UDP
    - 인터넷 계층 : 데이터 전송을 위한 주소 지정, 경로 설정
        - IP, ICMP, IGMP, ARP, RARP
    - 네트워크 액세스 계층 : 실제 데이터인 프레임을 송수신하는 역할
        - 이더넷, IEEE 802, HDLC, X.25, RS-232C

## 주요 프로토콜

- FTP : 컴퓨터 사이의 파일을 주고받는 원격 파일 전송 프로토콜
- SMTP : 전자 우편 교환
- 텔넷 : 멀리 떨어진 컴퓨터 접속, 원격 사용 서비스
- SNMP : TCP/IP 네트워크 관리 프로토콜, 라우터나 허브 등 네트워크 기기의 정보 관리 시스템
- HTTP : 월드 와이드 웹에서 HTML 문서를 송수신하기 위한 표준 프로토콜
- TCP : 양방향 연결에 스트림 위주, 신뢰성있고 순서 제어, 오류 제어, 패킷 분실 시 투명성 보장
    - 가상 회선 방식
- UDP : 비연결형으로 단순하며 오버헤드 적고 속도가 빠르나 신뢰성이 떨어짐
    - 데이터그램 방식
- IP : 전송 데이터에 주소를 지정, 경로 설정
- ICMP : IP와 조합하여 통신중에 발생하는 오류의 처리와 전송 경로 변경
- IGMP : 멀티캐스트를 지원하는 호스트나 라우터 사이 멀티캐스트 그룹 유지
- ARP : IP 주소를 MAC주소로 변경
- RARP : MAC 주소를 IP 주소로  변경
- 이더넷 : CSMA/CD 방식의 LAN
- L2TP : PPTP와 L2F 장점 결합 프로토콜, 자체적 암호화가 없어 다른 프로토콜과 함께 사용

## 네트워크 신기술

- 클라우드 컴퓨팅 : 컴퓨팅 자원을 중앙에 두고 인터넷으로 언제 어디서나 컴퓨터 작업
- 그리드 컴퓨팅 : 지리적 분산 컴퓨터를 초고속 인터넷망으로 연결, 하나의 고성능 컴퓨터
- 메시 네트워크 : 차세대 이동통신, 홈네트워킹
- SDN : 네트워크를 컴퓨터처럼 모델링하여 여러 사용자가 각각 소프트웨어로 네트워킹 가상화
- 피코넷 : 여러 개의 독립된 통신장치가 블루투스 기술로 통신망 형성
- 애드 훅 네트워크 : 재난현장같은 고정 유선망 없는 장소에서 모바일 호스트만으로 구성 네트워크
- 파장 분할 다중화 : 광섬유 통신기술로 파장 서로 다른 복수 신호를 보내 여러 단말기가 동시 회선
- 소프트웨어 정의 데이터 센터 : 데센의 모든 자원 가상화, 인력 없이 소프트웨어만으로 제어
- 개방형 링크드 데이터 (LOD) : 누구나 사용할 수 있는 웹상 공개 연계 데이터
- SSO (Single Sign On) : 한 번의 로그인으로 모든 사이트를 이용
